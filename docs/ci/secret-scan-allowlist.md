# Secret-scan allowlist hygiene playbook

This document is the operational companion to `.gitleaksignore` at the
repository root. `.gitleaksignore` is the data — one fingerprint per line
that the secret scanner is instructed to ignore. This file is the policy:
who can add entries, what must accompany them, how often they are audited,
and when they get removed.

The same policy applies to any equivalent waiver mechanism the scan suite
might grow in the future (e.g., a TruffleHog path-exclusion or a Trivy
`.trivyignore`). Where the mechanics differ, they are called out below.

## When the secret scanner fires

The first move is never "add an entry." Work through this checklist in
order:

1. **Treat the value as a real credential until proven otherwise.**
   Rotate it immediately if there is any doubt — at the issuing provider,
   in your password manager, in deployed secrets, everywhere. Rotation
   is cheap; an unrotated leaked credential is not.
2. **Remove or redact the value at source.** Move it to a fixture
   generator, a test helper, an environment variable, or a placeholder
   string the scanner does not match. The best waiver is no waiver.
3. **Only if removal is impossible**, fall through to the allowlist
   path. Cases where this applies in practice:
   - A deliberately-shaped synthetic fixture used to prove the scanner
     itself fires (the `tests/fixtures/secrets/` family).
   - An example token in user-facing documentation that must remain
     literal to be illustrative.
   - A historical commit that contains a value already rotated, where
     a rewrite would be more disruptive than the waiver.

## What every allowlist entry must include

A PR that adds an entry to `.gitleaksignore` must satisfy **all** of
the following before it can be merged:

- **Real-value certainty.** A statement in the PR description confirming
  the flagged value is not a live credential, with the evidence (e.g.,
  "this is `tests/fixtures/secrets/synthetic_aws_key.txt`, a fixture
  the round-trip test relies on; the value was generated by hand and
  has never been issued by AWS").
- **A removal attempt.** A statement that step 2 above was tried and
  why it cannot be applied here.
- **A documented owner.** The PR is reviewed by someone in the
  `security` code-owners group. Where ownership is ambiguous, the PR
  author proposes an owner and the reviewer confirms.
- **An inline comment.** The fingerprint line in `.gitleaksignore` is
  preceded immediately by a comment block naming:
  - the file the entry waives (relative path inside the repo);
  - the rule id (e.g., `aws-access-token`);
  - the rationale, in one sentence;
  - the date the entry was added (ISO 8601);
  - the owner who reviewed it.

A minimal example:

```text
# tests/fixtures/secrets/synthetic_aws_key.txt — aws-access-token
# Deliberately-shaped synthetic AWS access key used by the secret-scan
# round-trip test; not a real credential. Added 2026-05-14 by @ajbrown.
tests/fixtures/secrets/synthetic_aws_key.txt:aws-access-token:8
```

## Audit cadence

The allowlist is reviewed on a regular cadence so entries do not silently
outlive their justification.

- **Quarterly review.** A code-owner of `docs/ci/` opens an issue listing
  every active entry in `.gitleaksignore` and reaches out to the original
  reviewer (per the inline comment). For each entry the review decides
  one of: keep, update the comment, or delete.
- **Trigger-based review.** Whenever a referenced fixture, test, or
  document is renamed, refactored, or removed, the matching entry must
  be deleted in the same PR. A stale entry pointing at a file that no
  longer exists is a hard block on the next quarterly review.
- **Scanner-upgrade review.** When gitleaks (or any other secret
  scanner) is bumped to a new major version, every entry is re-validated
  against the new ruleset, because rule ids and fingerprints can shift
  between versions.

## What scans use the allowlist

`.gitleaksignore` is consumed by every gitleaks invocation in the
project:

- The pre-commit hook (`gitleaks` registered in
  `.pre-commit-config.yaml`) reads the file at the repo root.
- The `secret-scan.yml` workflow's `gitleaks` job reads the same file.
- `scripts/test-secret-scan.sh` exercises both halves of the
  round-trip: a run with an empty ignore file (asserts the scanner
  fires) and a run with a temporary file listing the synthetic
  fingerprint (asserts the allowlist is honoured).

If a contributor adds a new gitleaks invocation that does not pass
`--gitleaks-ignore-path=.gitleaksignore` (or its equivalent), that
invocation is wrong — it will diverge from the rest of the suite and
generate noise. Fix the new invocation, not the allowlist.

## Why TruffleHog does not share this file

TruffleHog uses `--only-verified` to suppress unverified findings, so
the "false positive" surface area is much smaller than gitleaks's. When
a verified-but-known-legitimate value does appear (rare during plugin
churn), the waiver path is `--exclude-paths` in the TruffleHog action's
`extra_args` rather than an entry here. The PR-time hygiene rules
above apply equally; the mechanics differ because TruffleHog has no
fingerprint allowlist.

If the TruffleHog waiver surface becomes routine, a future revision of
this document will formalize a separate `truffle-exclude.txt`-style
file and the same playbook will cover both.

## Adding a new allowlist mechanism

Before introducing any other allowlist (a new scanner's exclude file,
a custom rule suppression, etc.), update this document with:

1. the mechanism's location and format,
2. how PRs adding entries must justify them,
3. who audits the entries and how often,
4. how the mechanism interacts with the existing ones (does it apply
   pre-commit? in CI? both?).

The goal is one place a reviewer can look to understand every waiver
the scan suite carries.
