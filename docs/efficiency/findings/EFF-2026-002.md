## Evidence

This finding is a **seed entry** for the catalog — it documents the
`PM-CONNECTION-CHURN` pattern from PRD §18.9 and matches the shape that
`ConnectionChurnAnalyzer` (see
`crates/barista-netanalyze/src/analyzer/connection_churn.rs`) will emit from real
captures in M B.1 T3. The numbers below are representative of HTTP/1.1 behavior
against `repo.maven.apache.org` from Maven 3.9.x; they are not drawn from a specific
HAR file.

- Maven 3.9.x defaults to HTTP/1.1 with a small connection pool per host. A typical
  reactor that touches Maven Central for ~120 artifacts in a cold build opens **10
  or more fresh TCP/TLS connections** to `repo.maven.apache.org` even though every
  connection terminates at the same TLS endpoint and could be multiplexed over one
  HTTP/2 stream.
- Each fresh handshake costs one TCP RTT plus one TLS handshake (typically 2 RTT
  on TLS 1.2, 1 RTT on TLS 1.3) plus the handshake bytes. The combined wire
  overhead per handshake is on the order of 1.5–2 KiB.
- Same pattern reproduces against Sonatype Nexus 3.x and JFrog Artifactory hosts
  when the build tool is HTTP/1.1-only.

## Impact estimate

Rough estimate per representative cold build against a single repo host:

- **9 connections saved per build** (collapsing 10 H/1.1 connections to a single
  H/2 connection per host).
- ~2 KiB of TLS-handshake overhead per avoided connection ≈ **18 KiB wire saved
  per build**. This is a conservative figure — full handshakes with OCSP stapling
  and complete certificate chains can push it considerably higher.
- Latency savings are not captured by the byte/request/connection counters but are
  real: each avoided handshake is one to three RTT off the critical path, and on
  high-latency CI runners this dominates wall-clock time for the resolve phase.
- At ecosystem scale, this is the load-shedding-class concern: the §18.2
  "kindness to upstream" principle calls out connection count as the metric repo
  operators feel most acutely at peak. A single client converting from H/1.1 to
  H/2 reduces peak concurrent connections against Central by roughly the
  connection-pool size per build.

## Proposed mitigation

Implement **O-PROTO-01** from PRD §18.5: default to HTTP/2 for every repository
host, falling back to HTTP/1.1 only when the host explicitly negotiates it away
(no ALPN or explicit HTTP/1.1 in the response). This is unilateral and is wholly
within the HTTP-client stack the resolver chooses (`reqwest` / `hyper` already
support HTTP/2; the change is a single connection-builder flag).

Compose with **O-PROTO-02** (connection coalescing across hosts with shared
certificates) for cases where a repo manager fronts multiple logical repositories
behind a single wildcard cert. **O-REQ-09** (HTTP/2 multiplexing for metadata
fetches) completes the picture by ensuring the surviving connection actually
carries multiplexed streams rather than being used serially.

This is in M B.2 scope. The integration test for the analyzer in
`crates/barista-netanalyze/tests/fixtures/connection_churn_positive.har` already
exercises the detector against a synthetic HAR; the resolver change closes the
loop.

## Resolution

**Status: resolved (fix in code; ecosystem-scale byte/connection savings not yet
capture-measured, so not `proven`).**

`O-PROTO-01` is implemented in the upstream HTTP client — `Fetcher` in
`crates/barista-cache/src/fetch.rs`:

- **HTTP/2 by default.** With `http2_enabled` (the default) the client enables
  `http2_adaptive_window`, so it negotiates `h2` via ALPN over TLS and a single
  multiplexed connection carries the whole fetch fan-out. This is
  ALPN-negotiated, not prior-knowledge, so it falls back to HTTP/1.1 against
  HTTP/1.1-only hosts exactly as the mitigation requires.
- **Persistent connections.** The client sets an explicit `pool_idle_timeout`
  (90s) on top of `pool_max_idle_per_host`, keeping connections warm across a
  build's resolve burst so artifacts reuse one connection per host instead of
  churning fresh TCP/TLS handshakes.

Verification:

- `crates/barista-cache/src/fetch.rs: default_config_prefers_http2` — HTTP/2 is
  the default preference.
- `crates/barista-cache/src/fetch.rs: sequential_fetches_reuse_pooled_client` — a
  burst of fetches rides the shared pooled client.
- `crates/barista-cache/src/fetch.rs: concurrent_connection_ceiling_is_enforced`
  (pre-existing) — the pool bounds *simultaneous* connections under the ceiling,
  the connection-count guarantee this finding cares about.

Actual HTTP/2 stream negotiation is handled by `reqwest`/`hyper` (ALPN), not
Barista logic, so it is not unit-tested here (it needs an `h2`-capable TLS
server). Promotion to `proven` — measured connections-saved deltas vs. the
`mvn 3.9.x` baseline — happens via the efficiency-dashboard capture work
(`O-PROTO-01` connection-count signal from real HARs).
