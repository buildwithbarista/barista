## Evidence

This is a seed entry for the catalog covering **O-REQ-02** from PRD §18.3
(conditional fetches for `maven-metadata.xml`). The pattern is closely related
to `EFF-2026-001` (in-session dedup) but addresses a different waste mode:
even when a `maven-metadata.xml` fetch survives the in-session dedup check
(the first fetch of a given GA in an invocation), it commonly hits an
upstream server that already returned the same bytes minutes earlier — the
local cache has them but Maven 3.9.x's transport does not always send
`If-None-Match` / `If-Modified-Since`, so the bytes come down the wire again.

- A representative cold build resolving Spring Boot starters issues 8–12
  metadata GETs against Maven Central. With a warm local cache, all of
  them would be eligible for a 304 Not Modified roundtrip; in practice
  Maven's transport sends one but not always both validators, and on
  some repo managers a missing `If-Modified-Since` defeats the 304 path.
- The same fetch path observed against Sonatype Nexus 3.x returns full
  payloads when only `If-None-Match` is sent because the manager's ETag
  generation isn't byte-stable across upstream re-syncs.

**The captured shape does not exercise O-REQ-02.** O-REQ-02 only fires on

### Estimated impact (warm-cache CI shape)

The per-build numbers above are zero on the cold-cache shape, but the
warm-cache CI shape — the actual steady-state case in production — exhibits
the redundancy this finding documents:

- A representative `maven-metadata.xml` payload is on the order of 20 KiB
  uncompressed, ~5 KiB gzip. A 304 response is ~200 bytes.
- Eight surviving (post-dedup) metadata fetches that *could* return 304
  instead of a full payload save roughly 8 × (5 KiB − 0.2 KiB) ≈ **35 KiB
  wire per build** in the warm-cache case. Per-build request count is
  unchanged (the round-trip still happens); the byte savings dominate.
- The per-build figure scales linearly with the unique GA count touched
  in resolution. Monorepos and reactors with 30–60 distinct GAs see
  proportionally larger savings.

## Proposed mitigation

Implement **O-REQ-02** from PRD §18.3 — always send both `If-None-Match`
(when an ETag is known) and `If-Modified-Since` (when a Last-Modified is
known) on every `maven-metadata.xml` fetch. The cache layer already stores
these headers in its index entry (`crates/barista-cache/src/index.rs`); the
fetcher constructs the conditional headers from them
(`crates/barista-cache/src/fetch.rs::ConditionalHeaders`). The
B.2 T1 wiring exposes a per-session O-REQ-02 counter
(`oreq_02_avoided`) that advances when the underlying source reports
`FetchOrigin::Disk` or `FetchOrigin::InMemory` — the in-session proxy for
"the cache layer's conditional revalidation succeeded."

In-session, this composes with **O-REQ-01** (dedup at the resolver layer)
so the first surviving metadata fetch in a session is cheap (304 round-trip)
and every subsequent reference to the same GA is free (zero round-trips).

Unilateral; lives in `barista-cache`'s fetch path with the counter exposed
through `barista-resolver`'s `OreqSession`. The B.2 T1 implementation
attaches a per-session counter; B.2 T2 measures it against captured baselines.
