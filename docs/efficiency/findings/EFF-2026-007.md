## Evidence

This is a seed entry for the catalog covering **O-REQ-05** from PRD §18.3
(effective POM caching across invocations). The effective POM is the
result of running the (parent merge + interpolation + depMgt-application +
profile-activation) pipeline against a raw POM. The pipeline's inputs are
hash-stable: raw POM bytes + parent chain hashes + settings fingerprint
+ active profile set. Given identical inputs, the output is byte-identical.

- A representative effective-POM computation reads its raw POM plus 4–5
  parent POMs from cache (assuming O-REQ-04 fired) and produces a
  resolved `ResolvedPom` value. The compute cost is bounded, but the
  *fetch* cost on a cold cache is 4–5 POM GETs against upstream.
- In a multi-invocation CI scenario (`barista resolve` followed by
  `barista compile` in the same workspace), Maven 3.9.x re-runs the
  pipeline from scratch each time — there is no on-disk cache for the
  pipeline's output. Each invocation re-walks the parent chain.
- Same observation applies in-session: two paths in the BFS that arrive
  at the same `(coord, version)` would each re-run the pipeline if
  there were no in-session memoization.

| Bytes saved by effective-POM cache    | 0               | 0                            | 0     | 0               | 0                            | 0     |

**The in-session fire-surface is empty on these corpus shapes.** The two
paths in a BFS walk that *would* arrive at the same `(coord, version)` are
collapsed by the walker's nearest-wins before `fetch_pom` is called, so the
in-walk fire-surface for O-REQ-05 is empty in the single-pass walker that
v0.1 ships (M2.1 T2). The reachable v0.1 surface is the **cross-walk shared
session** — a CLI invocation that runs multiple `walk` calls against one
shared `OreqSession`. P02 and P03 run a single walk each, so this surface
is also empty for these captures.

The across-invocation surface — where O-REQ-05 pays off most decisively —
is M2.3's on-disk effective-POM cache; that is out of scope for B.2 T2 and
intentionally deferred until the cache pipeline lands.

The counter advances correctly on the cross-walk fixture — proving the

The in-walk surface awaits M2.3's parent-resolver path; the across-invocation
surface awaits the M2.3 on-disk cache.

### Estimated impact (cross-invocation workspace shape)

- 4 effective POMs re-resolved per invocation in a representative
  Spring Boot setup → **4 requests/build saved** at the parent-chain
  layer (effective-POM hit subsumes parent-POM and raw-POM fetches).
- POM byte size ranges 8–20 KiB uncompressed (~2–5 KiB gzip), so 4
  effective-POM hits save roughly **~50 KiB wire/build** in the cold
  case.
- The compute saving (parent-merge + interpolation + depMgt application
  is a non-trivial Rust workload — quick-xml parsing dominates) is in
  the same order of magnitude but is captured under the §18.6
  `O-COMP-*` family, not this finding.
- Across-invocation savings dwarf the in-session ones. A workspace
  that runs `resolve`, `compile`, `test`, `package` in sequence sees
  the effective-POM cache hit on the second and subsequent invocations,
  collapsing 4N round-trips to N over the workspace's lifetime.

## Proposed mitigation

Implement **O-REQ-05** from PRD §18.3. Two layers:

1. **In-session.** When the walker resolves `(coord, version)`'s
   effective POM for the first time, the session caches the
   `ResolvedPom`. Subsequent walks of the same `(coord, version)` in
   the same session serve from cache. This is the layer that
   B.2 T1 lands.
2. **Across-invocation.** A hash-stable on-disk cache keyed on
   `(raw_pom_sha256, parent_chain_sha256, settings_fingerprint,
    active_profile_set)` lives in `barista-cache` and serves cached
   effective POMs to subsequent invocations. This is M2.3 / B.2 T2's
   territory — B.2 T1 only lands the in-session piece.

The B.2 T1 implementation exposes a per-session counter
(`oreq_05_avoided`) that advances each time
`OreqSession::lookup_effective_pom` hits. The fire surface in the
current single-pass BFS walker is across-walk (a CLI invocation that
runs multiple `walk` calls against one shared session); the in-walk
surface activates once M2.3's parent-resolver chain runs through the
same session.

Unilateral. B.2 T1 lands the resolver wiring + counter; B.2 T2 measures
the in-session savings against captured baselines; M2.3 lands the
on-disk cache and the across-invocation savings.
