## Evidence

This finding is a **seed entry** for the catalog — it documents the
`PM-MISSED-COMPRESSION` pattern from PRD §18.9 and matches the shape that
`UncompressedTransferAnalyzer` (see
`crates/barista-netanalyze/src/analyzer/uncompressed_transfer.rs`) will emit from
real captures in M B.1 T3. The numbers below are representative of HTTP/1.1
behavior against repo-manager REST endpoints that return JSON metadata; they are
not drawn from a specific HAR file.

- Maven 3.9.x's default HTTP transport sends `Accept-Encoding: gzip, deflate` for
  artifact and metadata GETs against Maven Central, but several repo-manager REST
  APIs (Sonatype Nexus 3's `/service/rest/v1/search`, JFrog Artifactory's AQL
  endpoints) are sometimes hit by Maven plugins (versions-plugin,
  dependency-check-maven, OWASP scanners) without an `Accept-Encoding` header —
  the response comes back uncompressed even though the server supports gzip.
- A representative JSON search response for a moderately popular library against
  Nexus 3 is 180–220 KiB uncompressed, and 25–35 KiB gzip-compressed (typical
  text/JSON ratio of ~7×).
- Same pattern reproduces against Artifactory's AQL responses and against
  GitHub Packages' GraphQL endpoint when the client transport doesn't advertise
  encoding support.

## Impact estimate

Rough estimate per representative build that hits one of these APIs:

- A 200 KiB uncompressed JSON response served as 35 KiB gzip is a saving of
  **~140 KiB per affected response**. Round figures here — actual savings vary
  with the JSON shape (deeply repeated keys compress much better than
  artifact-coordinate lists do).
- Many builds hit zero such responses (resolver-only invocations don't touch the
  search APIs). Builds that run a dependency-version scan or an SCA check
  typically issue 1–3 such requests, so the per-build figure for *affected*
  builds is on the order of 140 KiB to a few hundred KiB.
- Zero request savings, zero connection savings — the inefficiency is purely
  payload-shaped. The §18.2 framing is about being polite to upstream by not
  pulling more bytes than necessary; uncompressed transfer is the cleanest
  example of accidental impoliteness.

## Proposed mitigation

Implement **O-XFER-02** from PRD §18.4: negotiate the best content encoding the
remote supports, transparently to the resolver and to plugins. The Barista HTTP
client should always send `Accept-Encoding: zstd, br, gzip` (in that preference
order — see PRD §18.4 for the rationale on zstd-first against Nexus 3.71+), and
should decompress the response in-stream so that callers receive a normal byte
stream with no awareness of the transport encoding.

Lower-effort first cut: just guarantee that *every* outbound request carries
`Accept-Encoding: gzip` — this matches what Maven 3.9.x already sends for
artifact/metadata GETs and closes the regression on the search-API path. The
zstd/brotli step is a follow-up once the resolver is HTTP/2 by default
(EFF-2026-002's mitigation).

Unilateral; lives in the cache crate's HTTP-client construction. M B.2 scope.

## Resolution

**Status: resolved (gzip baseline landed; zstd/brotli is a tracked follow-up).**

The shared upstream HTTP client — `Fetcher` in
`crates/barista-cache/src/fetch.rs`, reused for every artifact, POM, and
`maven-metadata.xml` GET — now negotiates gzip. The `reqwest` client is built
with gzip support enabled (`ClientBuilder::gzip(true)`), so every request that
does not set its own `Accept-Encoding` advertises `Accept-Encoding: gzip` and
the response is inflated in-stream; callers receive a plain byte stream with no
awareness of the transport encoding. This matches the Maven 3.9.x baseline and
closes the uncompressed-transfer regression on text resources (POM / metadata /
repo-manager JSON), which compress ~70–85%.

This implements the finding's recommended **first cut** (universal `gzip`). The
zstd-first / brotli preference order from PRD §18.4 is intentionally deferred to
follow the HTTP/2-default work (EFF-2026-002 / `O-PROTO-01`), per the "follow-up
once the resolver is HTTP/2 by default" note above — at which point the
`Accept-Encoding` preference list extends to `zstd, br, gzip`.

Verification (fix landed, **not yet** capture-measured, so not `proven`):

- `crates/barista-cache/src/fetch.rs: gzip_response_is_transparently_decompressed`
  — a `Content-Encoding: gzip` response is returned to the caller as the
  inflated plaintext.
- `crates/barista-cache/src/fetch.rs: requests_advertise_gzip_accept_encoding`
  — every outbound request carries `Accept-Encoding: gzip`.

Promotion to `proven` (with measured byte-download deltas against the `mvn 3.9.x`
baseline on the corpus) happens via the efficiency-dashboard capture work.
