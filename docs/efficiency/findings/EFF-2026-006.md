## Evidence

This is a seed entry for the catalog covering **O-REQ-04** from PRD §18.3
(parent POM deduplication). In a reactor build, every module under the
same parent re-resolves the parent chain independently. For
Spring-Boot-shaped projects this chain commonly is
`spring-boot-starter-X → spring-boot-starters → spring-boot-dependencies
→ spring-boot → spring-boot-parent` — 4–5 hops, each a separate GET
against the upstream repository, repeated once per module.

- A 14-module reactor each consuming `spring-boot-starter-web` re-fetches
  the 4–5-element parent chain per module, even though every chain
  resolves to the same bytes.
- Maven 3.9.x's `DefaultDependencyResolver` does carry an in-process
  POM cache, but it is scoped per `RepositorySystemSession` and several
  plugin paths construct a fresh session, defeating it.
- Same pattern reproduces in plugin resolution: a plugin's POM and its
  parent chain are re-fetched per *plugin invocation* in some
  configurations.

## Impact estimate

- A typical Spring Boot parent POM is 8–20 KiB uncompressed (~2–5 KiB
  gzip). A 5-element chain is 40–100 KiB uncompressed total.
- A 14-module reactor re-fetching that chain → 13 redundant chain walks
  → 65 redundant POM GETs → roughly **8 requests/build saved per chain**
  after collapsing intermediate cache hits, **~80 KiB wire/build** in
  the warm-cache-miss case.
- The number scales with both reactor module count and parent chain
  depth. Monorepos with 50+ modules and 6-level chains see savings 5×
  larger.
- Note: dependency-graph correctness is unaffected — every chain walk
  produces identical bytes, so collapsing them to one is byte-identical
  output with fewer round-trips.

## Proposed mitigation

Implement **O-REQ-04** from PRD §18.3. The resolver maintains an
in-session map from `(coord, version)` to fetched `RawPom`. When the
parent-resolver path (currently scaffolded as `NoParentResolver` in
`barista-resolver/src/walker.rs`; full implementation in M2.3) asks for
a parent POM, the session cache returns the cached value if present.

The same map covers any `fetch_pom` call — not just parent POMs — so
"sibling deps that resolve to the same coord+version" also benefit. In
the current single-pass BFS walker, nearest-wins kills duplicate visits
before they reach `fetch_pom`, so the immediate fire-surface for
O-REQ-04 within a single walk is small. The cross-walk dimension —
where a CLI invocation runs multiple walks against one shared session
(reactor builds, multi-module resolves) — is where the optimization
pays off in v0.1. The session is reused across walks, so the second
walk's shared transitive POMs are served from the cache.

The B.2 T1 implementation exposes a per-session counter
(`oreq_04_avoided`) that advances each time
`OreqSession::lookup_parent_pom` hits. The counter is also incremented
implicitly on effective-POM cache hits (`OreqSession::lookup_effective_pom`),
since an effective-POM hit subsumes a raw-POM hit.

Unilateral; lives entirely in `barista-resolver`. B.2 T1 lands the
in-session map and the walker wiring; M2.3's parent-resolver path
will use the same seam to dedup full chain walks.
