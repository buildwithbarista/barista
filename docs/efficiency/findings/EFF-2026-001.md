## Evidence

This finding is a **seed entry** for the catalog — it documents the
`PM-REDUNDANT-METADATA` pattern from PRD §18.9 and matches the shape that
`MetadataOverFetchAnalyzer` (see `crates/barista-netanalyze/src/analyzer/metadata_overfetch.rs`)
will emit once real captures from M B.1 T3 land. The evidence below is a representative
sketch drawn from manual `mvn -X` inspection of Spring Boot starter resolution, not from a
specific HAR file.

- A reactor resolving a Spring Boot starter set typically references 6–12 distinct
  GAV coordinates that share the GA `org.springframework.boot:spring-boot-starter`.
  Maven 3.9.x's default resolver issues one `maven-metadata.xml` GET per GAV against
  the configured remote repository even though the underlying GA-level metadata is
  identical across the group.
- In a representative single-invocation capture, three of those GETs are redundant —
  the response body, ETag, and Last-Modified are byte-identical to the first
  response, and they arrive within milliseconds of one another on the same TCP
  connection.
- Same behavior reproduces against Maven Central, Sonatype Nexus 3.x repos, and
  JFrog Artifactory. The redundancy is a client-side artifact of Maven's resolver,
  not a repo-server quirk.

**Key result: zero in-session metadata-fetch redundancy in the captured shape.**
Both corpus entries are single-module Spring Boot projects with fixed-version
dependency declarations and no `LATEST` / `RELEASE` / range coordinates. Under
cold-cache `mvn clean install -DskipTests`, mvn 3.9.x's resolver does not issue
`maven-metadata.xml` GETs at all — version-range resolution is the trigger, and
neither P02 nor P03 exercises it.

The barista cell delta (−7 / −14 entries) is the lifecycle-phase difference
between the upstream cell (`mvn clean install`) and the barista cell
(`barista verify --no-daemon` → `mvn verify clean`). The `install` phase fetches
a few additional plugin artifacts (e.g. `maven-install-plugin`) that `verify`
skips; that delta is not an O-REQ-01 effect.

The counter is zero by construction at v0.1: `barista verify --no-daemon` forks

the HAR. The counter activates against this corpus once `barista pull`'s
full-fetch path (or `barista verify`'s daemon path) routes resolution through
`barista-resolver::walker::walk` — M3.x cache-pipeline work.

### Estimated impact (ecosystem-scale)

The per-build numbers above are zero on this corpus shape, but the
ecosystem-scale story below is unchanged by that — it depends on the long
tail of consumers whose builds *do* trigger the redundancy:

- `maven-metadata.xml` for a popular GA is on the order of 20 KiB uncompressed
  (around 4–6 KiB with gzip, which most repo managers serve when asked).
- Three redundant fetches per build per affected GA at 20 KiB ≈ **60 KiB wire saved
  per build** in the uncompressed case. The number is roughly proportional to the
  number of GAVs in the resolution graph that share a GA — for monorepos and
  multi-module reactors with deep dep graphs it can be considerably higher.
- Request-count savings dominate the ecosystem-scale story: at Maven Central's
  published request volume, eliminating one redundant metadata GET per build per
  GA across the long tail of consumers is a multi-billion-request-per-month
  reduction (the §18.11 ecosystem report quantifies this; for v0.1 we treat the
  per-build number as the actionable signal).
- Connection savings: zero — the redundant fetches reuse the same connection
  Maven already opened, so the gain is purely in request count and bytes, not in
  TCP/TLS handshake count. (Connection churn is the separate `EFF-2026-002`
  story.)

## Proposed mitigation

Implement **O-REQ-01** from PRD §18.3: session-scoped per-`(repo, groupId, artifactId)`
deduplication of metadata fetches. The resolver maintains an in-memory map keyed on
that triple; the first request populates the map, every subsequent request for the
same key reads from the map instead of hitting the network. Cache lifetime is the
single CLI invocation — we deliberately do **not** persist this cache across
invocations to avoid serving stale metadata when the user has explicitly re-invoked
the resolver.

Companion fix: combine with **O-REQ-02** (conditional fetches via `If-None-Match` /
`If-Modified-Since`) so that the *first* fetch in a session is also cheap when the
upstream metadata hasn't changed since the last invocation. The two optimizations
compose — O-REQ-01 collapses N requests to 1, O-REQ-02 collapses the surviving
request to a 304-conditional round-trip when the metadata is stable.

This is unilateral (no upstream cooperation needed) and lives entirely in the
`barista-resolver` crate. Workstream B M B.2 will pick it up.
