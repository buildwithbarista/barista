## Evidence

This is a seed entry for the catalog covering **O-REQ-03** from PRD §18.3
(lockfile-mode skip metadata). In steady-state CI, the build pins every
resolved version via a lockfile (Maven plugin `versions:lock`, Gradle's
`gradle.lockfile`, or barista's own lockfile). When a lockfile pin is
present, fetching `maven-metadata.xml` to ask "which versions exist?" is
work the resolver doesn't need to do — the pin is authoritative.

- Maven 3.9.x re-fetches `maven-metadata.xml` per dependency on every CI
  run when `<dependency><version>` is a meta-version (`LATEST`/`RELEASE`)
  or a hard range. In typical Spring Boot reactor builds 8–14 such
  declarations exist (`spring-boot-starter-parent` pinning, BOMs that
  re-export ranges).
- The bytes are wasted twice: once to fetch the metadata, and once to
  serve the cached result the resolver would have used anyway given a
  valid pin.

## Impact estimate

- 12 metadata fetches per build is a representative count for a Spring
  Boot reactor with BOM-imported plugin versions and a few
  meta-version-declared dependencies. With a frozen lockfile:
  - **12 requests/build saved.**
  - Each metadata response is ~5 KiB gzip, so **~60 KiB wire/build**.
  - Connection count unchanged (the saved requests would have ridden
    the existing HTTP/2 connection); connection-level savings live
    in `EFF-2026-002`.
- At ecosystem scale, lockfile-mode is the most consequential
  request-count optimization — it's the steady-state CI shape and CI is
  where Maven Central sees the most concurrent thundering-herd load.

## Proposed mitigation

Implement **O-REQ-03** from PRD §18.3. When the resolver is invoked
under `--frozen` (or equivalent: a valid lockfile is present and
`--update` is not set), every metadata-fetch decision consults the
lockfile pin first:

- If a pin exists for the coord and the pin satisfies the declared
  requirement (soft, hard-range, `LATEST`, or `RELEASE`), use the pin
  directly and skip the metadata fetch.
- If no pin exists, fall back to the metadata-fetch path. This keeps
  partially-pinned lockfiles (a real-world case) working correctly.

For SNAPSHOT entries with stale TTL, the pin remains authoritative but
a per-version `maven-metadata.xml` fetch may still be needed to find the
timestamped publish. That subtree is out of scope for O-REQ-03 — it's
the SNAPSHOT-freshness path.

The B.2 T1 implementation introduces an `OreqSession::with_frozen_pins`
seam that `barista-resolver::walker` consults at the metadata-resolution
site. The per-session counter (`oreq_03_avoided`) advances each time the
short-circuit fires.

Unilateral; lives entirely in `barista-resolver` + the lockfile reader.
B.2 T1 lands the resolver wiring; the CLI-side `--frozen` flag is M4.3
territory and will plumb the pins through to `WalkOptions::oreq`.
